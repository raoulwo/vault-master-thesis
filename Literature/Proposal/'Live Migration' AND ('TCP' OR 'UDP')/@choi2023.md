### Daniel's Critique

- It's a classic *workshop* paper:
	- Good idea and problem statement
	- Evaluation isn't thorough
		- Comparison between Capybara (with kernel-bypass) and Prism (without kernel-bypass)
			- Of course Capybara is faster without the Linux kernel overhead
	- Stress tests are missing
		- How does the system function with increased number of TCP connections
		- How does the system act under different loads (high throughput vs. low throughput)
		- How does the system act using TLS (which requires more state to migrate)
	- Demikernel is created by Microsoft and the paper has Microsoft Research participation
		- Could definitely doubt whether the paper is impartial or not
		- That's the point where we can connect to the paper
			- Create experiments that show that Capybara is insufficient in aspects X and Y
				- Propose improvements, thus we have research challenges
			- Even though their paper works in context of datacenters
				- Possible that there aren't any papers about I/O migration for orbital edge
				- Can say that their cloud assumptions A, B, C don't apply in orbit
					- Therefore their strategy doesn't work in orbit and we have to extend it

### Idea

| Dimension | Capybara's Assumption (Data Center) | LEO Satellite Reality (Orbital Edge) |
|---|---|---|
| Network Fabric | Ultra-low, stable RTTs, near-zero packet loss | dynamic, fluctuating RTTs, high packet loss |
| Topology | Static single-rack system with a centralized ToR switch | Highly dynamic topology, constant handovers |
| Hardware | Centralized, line-rate programmable switches | Constrained onboard hardware, no ToR switch |

- Capybara gets 4 microseconds average migration time by using kernel-bypass and offloading packets to a specialized switch
- With increased packet loss or without specialized switch, the results might evaporate 

#### Proposal Skeleton

- Title: 
	- e.g. Evaluating Kernel-Bypassed Live TCP Migration in Lossy LEO Satellite Constellations
	- e.g. A Comparative Evaluation of Microsecond-Scale Live TCP Migration in Orbital Edge Computing Environments
- Introduction/Motivation:
	- Explain why orbital edge computing requires TCP migration
		- Maintaining state during frequent handovers
- Problem statement:
	- Critique Capybara
		- Unfair evaluation of kernel-bypass vs Linux kernel
			- Obscures true system overhead
	- Kernel-bypass works in ideal data center environments
		- Data center environment assumptions like programmable switch is infeasible in orbit
	- Performance, state synchronization, and two-phase commit are unproven under high packet loss and scale of LEO networks
- Research Questions and Hypothesis:
	- RQ1 (scale): How does Capybara's two-phase commit migration protocol scale when connection density drastically increases?
		- H1: Under orbital handover scenarios with massive concurrent connections, Capybara's reliance on centralized switch registers and destination-side transient packet buffering will lead to memory exhaustion and dropped connections.
	- RQ2 (state complexity): What happens when we increase layer-4 state complexity by introducing TLS?
		- H2: Migrating larger state (TCP state + TLS session keys and crypto) bottlenecks the userspace serialization phase, erasing the microsecond latency benefits
- Hypothesis: As network packet loss and connection density scale to LEO-relevant levels, performance benefits of kernel-bypassed live TCP migration degrade sharply due to state-synchronization timeouts and buffer exhaustion

#### Experiments

- Can't launch satellites, will need to focus on network emulation/simulation
- Outline testbed using tools like `ns-3`, `Mininet` and `NetEm` to inject packet loss and latency

Experiment 1 (actual baseline):

- Capybara unfairly compared kernel-bypass to standard Linux
	- If we strip away the unfair kernel-bypass advantage, how does Capybara actually perform under harsh conditions?
		- Capybara's evaluation is too shallow and might hide performance bottlenecks
			- What happens at scale?
			- What happens using real-world TLS?
		- Capybara's assumptions about the data center environment disintegrate in space
- Setup: Compare Capybara to a kernel-bypass optimized Prism stack to measure migration protocol overhead without kernel-bypass bias
	- Alternatively: Propose a control group introducing a standard kernel-bypass TCP stack without Capybara's switch-based logic, or evaluate standard Linux TCP repair under the exact same high-latency, high-loss parameters
- Measure the following metrics:
	- Determine if CPU efficiency of kernel-bypassing matters if packet loss is high

Experiment 2 (scale and throughput):

- Evaluate how Capybara's migration latency scales with variable packet loss
- Fix a constant workload and increase the packet loss
- Measure the following metrics:
	- Total migration time
	- $P_{99}$ tail latency
	- number of connection drops/resets caused by lost migration-handshake messages

Experiment 3 (packet loss):

- Capybara relies on a temporary buffer on destination server to hold packets during migration
- What happens if thousands of connections migrate at once?
- Keep packet loss fixed at a baseline, scale the number of TCP connections to simulate sudden handoffs (100, 10000, 100000 concurrent connections)
- Can also vary the traffic profiles (high throughput vs. low throughput)
- Measure the following metrics:
	- Memory usage of library OS transient buffers
	- Buffer overflow rates
	- Throughput drop during migration window 

Experiment 4 (TLS state bloat):

- Capybara only realizes standard TCP buffers, what happens if the migrated state size increases using TLS
- Vary the state size (standard TCP vs. TCP + TLS state)
	- Think of how to simulate it instead of actually implementing TLS
- Measure the following metrics:
	- Serialization/deserialization times on the host
	- resulting inflation of tail-latency

## Related Work

%% Existing state-of-the-art research. %%

Introduction:

- Modern kernel-bypassing network stacks provide sub-microsecond TCP processing
	- For data centers could lead to ultra-low median response time and predictable, microsecond-scale latency
	- Problem: Unpredictable workloads, traffic bursts, and dynamic resource allocation prevent applications from reaping the benefits of microsecond-scale data center hardware and networking stacks
- Recent research has made great strides in achieving predictable load balancing for microsecond-scale workloads
- Load balancers for TCP connections can only statically assign entire connections to backend servers, insufficient to ensure consistent microsecond-scale latency
- If we could efficiently migrate TCP connections, it would enable load balancers to dynamically assign servers
- TCP migration has been explored in the past: 9, 23
	- Problem: Latency overhead

Core question: Can we migrate TCP-connections at microsecond scale?

Motivation for microsecond-scale migrations:

- Workload skew can cause load imbalance across servers, leading to degraded response time
	- Especially important for strict latency SLOs
	- Load balancing issues need to be addressed efficiently
- Programmable switches for dynamic load balancing are promising
	- Existing works only work for single-packet sized requests over UDP
	- However, existing data centers often rely on TCP for
		- reliable messaging
		- bandwidth utilization
		- congestion control
	- Paper asks if they can do TCP-based load balancing

## Contributions

%% Research gap and novel contributions. %%

- Capybara, a TCP migration system for microsecond-scale migration latency
- In-network approach minimizes live migration impact and ensures TCP processing correctness

Three key design insights:

- Design of a microsecond-scale TCP stack
- TCP-stack leverages determinism to efficiently migrate state
- Use of programmable switches to monitor server load and providing transient state management

## Methodology

%% Architecture of the solution. %%

Design:

- Kernel-bypass TCP stack
- Load balancer implemented on a high performance programmable switch
- Deterministic TCP implementation based on Demikernel
- Two-phase commit protocol to migrate TCP connections

NOTE: System is specifically designed in context of datacenters, not LEO networks

Two main components:

- In-network load monitoring and migration management
- Two-phase TCP migration protocol

Programmable switch has two tasks:

- Maintaining original and current server address information
- Migration-aware dynamic packet forwarding

TCP Stack:

- Built on-top of Demikernel
- Must be atomic -> two-phase migration:
	1. Phase: prepare migration
	2. Phase: connection state transfer

Demikernel benefits:

- TCP migration in userspace leads to lower overhead and predictable latencies
- Userspace libraries are easier to develop
